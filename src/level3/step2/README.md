# 🦁 멋사 공주대 백엔드 객체지향 2주차 과제 - Level 3-2

## 과제 주제
같은 요청을 보내면 객체가 스스로 다르게 동작하도록 만들어,  
**다형성**의 필요성과 장점을 코드로 느껴보기

## 과제 목표
이번 단계에서는 Level 3-1에서 작성한 `if-else` 분기 방식을 개선하여,  
외부에서 타입을 일일이 검사하지 않고도 각 객체가 자신의 역할에 맞게 스스로 동작하는 구조를 구현합니다.

즉, 이번 단계의 핵심은 아래와 같습니다.

- 같은 부모 타입으로 여러 객체를 다룰 수 있다
- 외부는 공통된 메시지만 보내면 된다
- 실제 동작은 각 객체가 자신의 오버라이딩 메서드로 결정한다
- 분기문이 줄어들고 구조가 더 유연해진다

---

## 사전 준비
이번 단계 역시 Level 2에서 구현한 `Member`, `BabyLion`, `Staff`, `Leader`를 그대로 사용합니다.

이번 단계에서는 기존 객체들을 `Member[]`로 묶어 관리하며,  
조건문 대신 다형성을 활용해 더 유연하게 동작하도록 `ClubManager`를 구현하는 데 집중하세요.

---

## 상황 설명
Level 3-1에서는 `IfElseClubManager`가  
구성원들의 타입을 일일이 검사해서 행동을 분기 처리했습니다.

그 방식은 동작은 하지만, 새로운 역할이 추가될 때마다 분기문을 수정해야 하고,  
외부 클래스가 각 객체의 상세한 타입을 모두 알아야 한다는 문제가 있었습니다.

이번 단계에서는 그 구조를 개선합니다.

이제 관리 클래스는 더 이상 “이 객체가 아기사자인가? 운영진인가? 대표인가?”를 계속 묻지 않습니다.

대신 모든 객체를 `Member`로 보고, 그저 같은 요청을 보냅니다.

예를 들어 아래와 같은 방식입니다.

- 각 구성원에게 `doTask()`를 호출한다
- 실제 어떤 행동을 할지는 각 객체가 스스로 결정한다

즉, 외부는 메시지만 보내고 실제 동작은 객체 내부에 맡기는 구조로 바꾸는 것입니다.

---

## 구현할 내용
이번 단계에서는 `ClubManager` 클래스를 완성하세요.

`ClubManager`는 멋사 구성원 전체를 관리하며, 각 구성원에게 공통된 메시지를 보내는 역할을 합니다.

이때 중요한 점은, `ClubManager`가 각 자식 클래스의 구체적인 타입을 몰라도 동작해야 한다는 것입니다.

---

## 구현 조건

### 1. Member 배열로 객체 묶기
`BabyLion`, `Staff`, `Leader` 객체를 모두 `Member[]` 배열에 담아 관리하세요.

이 부분은 Level 3-1과 같지만, 이번에는 배열에 담긴 객체들을 처리하는 방식이 달라집니다.

### 2. 공통 메서드 호출로 역할 수행하기
`ClubManager`는 `Member[]`를 순회하면서 각 구성원에게 공통된 요청만 보내야 합니다.

예를 들어 아래와 같은 흐름이 가능해야 합니다.

- 이름 출력
- 역할 출력
- `doTask()` 호출

이때 `ClubManager`는 이 객체가 `BabyLion`인지 `Staff`인지 `Leader`인지 굳이 알 필요가 없습니다.

그저 `member.doTask()`를 호출하면 됩니다.

### 3. 실제 동작은 객체가 스스로 결정하기
외부에서 같은 `doTask()`를 호출하더라도, 실제 어떤 코드가 실행될지는 각 객체가 결정해야 합니다.

즉,

- `BabyLion` 객체라면 아기사자의 `doTask()` 실행
- `Staff` 객체라면 운영진의 `doTask()` 실행
- `Leader` 객체라면 대표의 `doTask()` 실행


### 4. 공통 동작과 예외적 동작 구분하기
대부분의 흐름은 공통 타입인 `Member` 기준으로 처리할 수 있습니다.

하지만 꼭 필요한 경우에는 특정 타입만 따로 처리할 수도 있습니다.

예를 들어 과제 제출은 아기사자만 하도록 만들 수 있습니다.

이 경우에는 `instanceof`를 사용할 수 있지만, 핵심 동작까지 모두 분기로 처리해서는 안 됩니다.

즉, 이 단계에서는 **주요 흐름은 다형성으로 처리하고, 예외적인 경우에만 제한적으로 타입 확인을 사용**하는 것이 중요합니다.

### 5. Level 3-1과 비교하기
이번 단계에서는 작성한 코드가  
Level 3-1의 `if-else` 방식보다 어떻게 더 깔끔하고 유연한지 직접 비교해보세요.

예를 들어 아래 질문을 떠올려볼 수 있습니다.

- 새로운 역할이 추가되면 어디를 수정해야 하는가?
- 관리 클래스가 각 타입을 모두 알고 있어야 하는가?
- 역할별 책임이 어디에 위치하는가?

---

## 왜 이렇게 설계할까?

### 다형성이 필요한 이유
아기사자, 운영진, 대표는 모두 `Member`이므로  
프로그램 입장에서는 하나의 공통 타입으로 다룰 수 있어야 합니다.

그리고 같은 요청을 보내더라도 실제 동작은 객체마다 달라야 합니다.

이 두 가지를 동시에 가능하게 해주는 것이 다형성입니다.

즉, 다형성을 사용하면 외부 코드는 단순해지고, 객체는 자신의 책임을 스스로 수행할 수 있습니다.

### if-else 방식보다 나은 이유
Level 3-1의 방식에서는 관리 클래스가 객체의 타입을 일일이 확인하고,  
각 타입에 맞는 행동을 직접 결정해야 했습니다.

하지만 다형성을 사용하면 외부는 공통 메서드만 호출하면 되고, 실제 행동은 객체 내부에 들어갑니다.

그래서
- 분기문이 줄어들고
- 새로운 역할이 추가되어도 기존 관리 코드를 덜 수정하게 되며
- 역할별 책임이 각 객체 안에 자연스럽게 모이게 됩니다

### 객체지향스러운 이유
객체지향에서는 보통 외부가 객체의 내부 구현을 세세하게 아는 것보다,  
객체에게 메시지를 보내고 객체가 책임에 따라 응답하도록 만드는 것이 더 자연스럽습니다.

`doTask()`는 이런 구조를 가장 잘 보여주는 예시입니다.

외부에서는 단지 `doTask()`를 호출할 뿐이지만, 실제 동작은 각 객체가 자신의 역할에 맞게 수행합니다.

---

## 실행 목표
이번 단계에서는 서로 다른 객체들을 `Member[]` 배열에 담고,  
`ClubManager`가 각 객체에게 같은 요청을 보내더라도 실행 결과가 객체마다 다르게 나타나야 합니다.

예를 들어 아래와 같은 흐름이 가능해야 합니다.

- `BabyLion`, `Staff`, `Leader` 객체 생성
- 이 객체들을 모두 `Member[]` 배열에 저장
- `ClubManager`가 배열을 순회
- 각 객체의 이름과 역할 출력
- 각 객체에 대해 `doTask()` 호출
- 필요한 경우 아기사자만 과제 제출 처리

---

## 예시 실행 결과
아래는 한 가지 예시입니다.

철수 / 역할: 아기사자
철수님은 세션을 듣고 학습합니다.
-------------------

영희 / 역할: 운영진
영희님은 세션 자료를 준비하고 발표를 진행합니다.
-------------------

민석 / 역할: 대표
민석님은 운영 현황을 파악하고 방향을 조율합니다.
-------------------

하늘 / 역할: 아기사자
하늘님은 세션을 듣고 학습합니다.
-------------------

철수님이 과제를 제출했습니다.
하늘님이 과제를 제출했습니다.

출력 문구는 완전히 같지 않아도 되지만, 핵심 흐름이 `if-else` 분기보다 더 단순하게 표현되어야 합니다.

## 생각해볼 질문
1. 왜 `ClubManager`는 `Member[]`만 알아도 동작할 수 있는가?
   1. `Member[]` 안의 객체들이 스스로 행동하기 떄문이다.
2. `member.doTask()`라는 같은 코드를 실행했는데 결과가 달라지는 이유는 무엇인가?
   1. 각각의 객체들별로 구분된 `doTask()`가 호출되어 다른 문구들이 불러와진다.
3. Level 3-1의 방식보다 이번 방식이 더 유연한 이유는 무엇인가?
    1. 3-1에서는 외부 클래스가 각 객체의 상세한 타입을 모두 알아야하는 문제가 있었습니다.
   2. 이번 방식은 외부는 매세지만 보내고 실제 동작은 객체 내부에게 맡겨, 유지보수가 더 유연합니다.
4. 새로운 역할 클래스가 추가되면 어떤 클래스만 추가 또는 수정하면 되는가?
   1. Main 클래스만 추가/수정하면 된다.
5. 다형성은 “외부 코드의 복잡도”를 어떻게 줄여주는가?
    1. 외부가 객체 내부의 구현을 세세하게 알지 않아도 된다.

## 힌트
- 자식 객체는 부모 타입 변수에 저장할 수 있습니다.
- 오버라이딩된 메서드는 실제 객체 기준으로 실행됩니다.
- 공통 흐름은 부모 타입 기준으로 처리하는 것이 핵심입니다.
- `instanceof`는 예외적인 경우에만 제한적으로 사용하는 것이 좋습니다.
- Level 3-1의 분기문과 현재 코드를 직접 비교해보면 차이가 더 잘 보입니다.
