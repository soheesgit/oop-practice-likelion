# 🦁 멋사 공주대 백엔드 객체지향 2주차 과제 - Level 3-1

## 과제 주제
`if-else` 분기 방식으로 멋사 구성원들의 행동을 처리해보며, 이 방식의 한계를 코드로 느껴보기

## 과제 목표
이번 단계에서는 `BabyLion`, `Staff`, `Leader` 객체를 하나의 배열에 담아 관리하되,  
각 객체의 역할 수행을 `if-else` 또는 `instanceof` 분기로 처리해봅니다.

이 단계의 목적은 단순히 동작하는 코드를 만드는 것이 아니라,  
**왜 이런 방식이 점점 불편해지는지 직접 느껴보는 것**입니다.

즉, 이번 단계의 핵심은 아래와 같습니다.

- 서로 다른 객체를 하나의 타입으로 묶을 수는 있다
- 하지만 실제 행동을 처리할 때 `if-else`가 많아지면 코드가 복잡해진다
- 역할이 늘어날수록 관리가 어려워진다
- 다음 단계에서 다형성이 왜 필요한지 자연스럽게 이해한다

---

## 사전 준비
이번 단계는 Level 2에서 구현한 `Member`, `BabyLion`, `Staff`, `Leader`를 그대로 사용합니다.

이번 단계에서는 구성원 클래스 자체를 새로 만드는 것이 아니라, 이 객체들을 관리하는 `IfElseClubManager`를 구현하며  
조건문 기반 처리 방식의 한계를 확인하는 데 집중하세요.

---

## 상황 설명
Level 2에서는 `BabyLion`, `Staff`, `Leader`를 각각 클래스로 만들고, 각 클래스가 `doTask()`를 오버라이딩하도록 구현했습니다.

하지만 이번 단계에서는 일부러 그 오버라이딩된 메서드를 바로 활용하지 않고, 외부의 관리 클래스가 객체의 실제 타입을 검사해서  
직접 분기 처리하는 방식을 먼저 사용해봅니다.

예를 들어 아래와 같은 방식입니다.

- 객체가 `BabyLion`이면 학습 관련 메시지 출력
- 객체가 `Staff`이면 운영진 관련 메시지 출력
- 객체가 `Leader`이면 대표 관련 메시지 출력

즉, 객체 스스로 행동하게 하기보다 외부에서 “너는 무슨 타입이니?”를 확인한 뒤 행동을 결정하는 방식입니다.

---

## 구현할 내용
이번 단계에서는 `IfElseClubManager` 클래스를 완성하세요.

`IfElseClubManager`는 멋사 구성원 전체를 관리하며,  
각 구성원의 실제 타입을 확인해서 역할 수행을 분기 처리하는 클래스입니다.

---

## 구현 조건

### 1. Member 배열로 객체 묶기
`BabyLion`, `Staff`, `Leader` 객체를 모두 `Member[]` 배열에 담아 관리하세요.

예를 들어 아래와 같은 흐름이 가능해야 합니다.

- `BabyLion` 객체 생성
- `Staff` 객체 생성
- `Leader` 객체 생성
- 이 객체들을 `Member[]` 배열에 저장

즉, 저장은 하나의 타입으로 하되, 처리는 분기문으로 직접 나누게 됩니다.

### 2. 타입 검사로 역할 수행 분기하기
`IfElseClubManager`는 `Member[]`를 순회하면서  
각 객체의 실제 타입을 확인한 뒤 역할 수행을 처리해야 합니다.

예를 들어 아래와 같은 구조가 가능합니다.

- `member instanceof BabyLion`
- `member instanceof Staff`
- `member instanceof Leader`

그리고 각 조건에 따라 다른 출력이나 동작을 수행합니다.

### 3. 외부에서 행동을 직접 결정하기
이번 단계에서는 객체 내부의 `doTask()`를 믿고 맡기기보다,  
관리 클래스가 직접 분기해서 행동을 결정하는 방식으로 구현합니다.

즉, 아래처럼 외부가 로직을 다 들고 있게 됩니다.

- 아기사자면 학습 메시지 출력
- 운영진이면 발표 메시지 출력
- 대표면 조율 메시지 출력

이 방식은 당장은 동작하지만, 구조적으로 어떤 불편함이 생기는지 직접 살펴보세요.

### 4. 과제 제출도 분기 처리하기
예를 들어 아기사자만 과제를 제출하게 만들고 싶다면, 이 역시 분기문을 통해 처리할 수 있습니다.



- `BabyLion`이면 `submitAssignment()` 호출
- 그 외는 아무 동작 없음

처럼 구현할 수 있습니다.

### 5. 새로운 역할 추가를 상상해보기
만약 새로운 구성원 역할이 하나 더 추가된다면, 이 관리 클래스에서 어떤 부분을 수정해야 할지 생각해보세요.

예를 들어 `Mentor` 같은 역할이 추가되면 기존 분기문에도 새로운 조건이 계속 추가되어야 할 수 있습니다.

---

## 왜 이렇게 설계할까?

### 이 단계를 먼저 하는 이유
다형성의 장점은 설명만 들으면 추상적으로 느껴질 수 있습니다.

그래서 먼저 `if-else` 방식으로 구현해보면,  
외부에서 타입을 계속 확인해야 하고,  
역할이 늘어날수록 분기문이 길어지며,  
기존 코드를 자꾸 수정해야 한다는 점을 직접 체감할 수 있습니다.

이 불편함을 느껴야 다음 단계의 다형성이 왜 더 좋은지 분명하게 이해할 수 있습니다.

### if-else 방식의 한계
이 방식에서는 객체가 스스로 행동하지 않고, 외부 관리자가 객체의 타입을 모두 알고 있어야 합니다.

즉,

- 객체가 늘어날수록 분기문이 길어지고
- 새로운 역할이 추가될 때마다 기존 코드를 수정해야 하며
- 역할별 책임이 객체 안이 아니라 외부에 흩어질 수 있습니다

이런 구조는 유지보수에 불리합니다.

### 객체지향스럽지 않은 이유
객체지향에서는 보통  
“객체에게 메시지를 보내고, 실제 동작은 객체가 스스로 결정하게 하는 것”이 자연스럽습니다.

하지만 `if-else` 방식은 외부가 객체의 정체를 모두 확인한 뒤  
행동까지 대신 결정해주는 구조입니다.

즉, 객체 중심이라기보다 조건문 중심으로 흐르게 됩니다.

---

## 실행 목표
이번 단계에서는 서로 다른 객체들을 `Member[]` 배열에 담고,  
`IfElseClubManager`가 타입 검사를 통해 각 역할을 분기 처리할 수 있어야 합니다.

예를 들어 아래와 같은 흐름이 가능해야 합니다.

- `BabyLion`, `Staff`, `Leader` 객체 생성
- 이 객체들을 `Member[]` 배열에 저장
- `IfElseClubManager`가 배열을 순회
- `instanceof` 또는 `if-else`로 타입 판별
- 타입별로 다른 메시지 출력
- 아기사자만 과제 제출 처리

---

## 예시 실행 결과
아래는 한 가지 예시입니다.

철수 / 역할: 아기사자
철수님은 세션을 듣고 학습합니다.
-------------------

영희 / 역할: 운영진
영희님은 세션 자료를 준비하고 발표를 진행합니다.
-------------------

민석 / 역할: 대표
민석님은 운영 현황을 파악하고 방향을 조율합니다.
-------------------

철수님이 과제를 제출했습니다.

출력 문구는 완전히 같지 않아도 되지만,  
역할별 처리가 `if-else` 또는 `instanceof` 분기문으로 이루어져야 합니다.

## 생각해볼 질문
1. `IfElseClubManager`가 각 자식 클래스의 타입을 모두 알아야 하는 이유는 무엇인가?
   1. 타입별로 출력되는 문구가 다르기 때문이다.
2. 새로운 역할이 하나 추가되면 어떤 코드가 수정되어야 하는가?
   1. 타입 확인 후 출력하는 코드를 추가해야한다.
3. 역할별 로직이 외부 클래스에 모이면 어떤 문제가 생길 수 있는가?
   1. 역할이 늘어날수록 분기문이 길어지고 하나가 추가되면 전부를 수정해야한다. 유지보수에 불리하다.
4. 지금 구조가 객체에게 일을 시키는 방식인지, 외부가 대신 판단하는 방식인지 설명해보기
   1. 외부가 대신 판단하는 방식이다. 외부에서 판단하여서 타입의 종류에 따라 다른 문구를 출력하기 때문이다.

## 힌트
- `instanceof`를 사용하면 객체의 실제 타입을 확인할 수 있습니다.
- `Member[]`에 담겨 있어도 실제 객체는 각각 다를 수 있습니다.
- 이번 단계에서는 일부러 분기문으로 처리하는 것이 목적입니다.
- 다음 단계에서 같은 문제를 다형성으로 개선하게 됩니다.
