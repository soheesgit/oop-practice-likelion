# 🦁 멋사 공주대 백엔드 객체지향 2주차 과제 - Level 2

## 과제 주제
멋사 구성원을 구체적인 클래스로 확장하며 **상속**의 필요성을 코드로 느껴보기

## 과제 목표
이번 단계에서는 Level 1에서 만든 `Member` 추상 클래스를 바탕으로,  
멋사의 실제 구성원인 아기사자, 운영진, 대표를 각각 클래스로 구현합니다.

즉, 아래 두 가지를 직접 코드로 경험하는 것이 목표입니다.

- **상속**: 공통 속성과 기능은 부모 클래스에서 재사용하고, 각 역할에 맞는 기능은 자식 클래스에서 확장하기
- **오버라이딩**: 같은 메서드 이름이라도 각 객체의 역할에 맞게 다르게 동작하도록 구현하기

---

## 상황 설명
Level 1에서는 멋사의 모든 구성원을 `Member`라는 공통 개념으로 추상화했습니다.

하지만 실제로 멋사에는 서로 다른 역할을 가진 구성원이 존재합니다.

- 아기사자: 세션을 듣고 학습하며 과제를 제출하는 사람
- 운영진: 세션 자료를 준비하고 발표 및 피드백을 하는 사람
- 대표: 운영 현황을 파악하고 동아리의 방향을 조율하는 사람

이들은 모두 `Member`라는 공통점을 가지지만,  
각자가 맡은 책임과 수행하는 행동은 서로 다릅니다.

이번 단계에서는 이러한 차이를 **상속 구조**로 표현합니다.

---

## 구현할 내용
아래 3개의 클래스를 구현하세요.

- `BabyLion`
- `Staff`
- `Leader`

이 클래스들은 모두 `Member`를 상속받아야 합니다.

---

## 구현 조건

### 1. 상속 사용
아래 세 클래스는 모두 `Member`를 상속받아야 합니다.

- `BabyLion`
- `Staff`
- `Leader`

### 2. 생성자에서 부모 생성자 호출
각 클래스는 이름을 받아 생성되며,  
부모 클래스인 `Member`의 생성자를 호출해야 합니다.

### 3. 역할 반환 메서드 구현
각 클래스는 자신의 역할을 문자열로 반환해야 합니다.

- `BabyLion` → `"학습자"`
- `Staff` → `"운영진"`
- `Leader` → `"대표"`

즉, `getRole()`을 각 클래스에서 오버라이딩해야 합니다.

### 4. 책임 수행 메서드 구현
각 클래스는 자신의 책임에 맞는 동작을 `doTask()`로 구현해야 합니다.

예시 출력 형식:

- `BabyLion` → `"<이름>님은 세션을 듣고 학습합니다."`
- `Staff` → `"<이름>님은 세션 자료를 준비하고 발표를 진행합니다."`
- `Leader` → `"<이름>님은 운영 현황을 파악하고 방향을 조율합니다."`

### 5. 각 클래스의 고유 기능 추가
각 클래스에는 자신의 역할을 더 잘 드러내는 **고유 메서드**를 1개 이상 추가하세요.

예시:
- `BabyLion` → 질문하기
- `Staff` → 피드백 주기
- `Leader` → 외부와 소통하기

메서드 이름과 출력 문구는 자유롭게 정해도 되지만, 역할에 맞아야 합니다.

---

## 왜 이렇게 설계할까?

### 상속이 필요한 이유
아기사자, 운영진, 대표는 모두 이름이 있고, 과제를 제출할 수 있으며, 역할과 책임을 가집니다.  
이 공통 요소를 각 클래스마다 중복해서 작성하면 코드가 비효율적이고 관리하기 어려워집니다.

그래서 공통 속성과 기능은 `Member`에 두고,  
각 역할에 따라 달라지는 부분만 자식 클래스가 구현하도록 만드는 것이 더 적절합니다.

### 오버라이딩이 필요한 이유
`doTask()`라는 같은 이름의 메서드를 사용하더라도,  
아기사자와 운영진과 대표가 해야 하는 일은 서로 다릅니다.

즉, 같은 메시지를 받더라도 객체마다 다르게 반응해야 하며,  
이를 코드로 표현하는 것이 오버라이딩입니다.

## 실행 목표
이번 단계에서는 각 객체를 직접 생성하고,  
부모 클래스에서 물려받은 기능과 자식 클래스에서 구현한 기능을 모두 확인할 수 있어야 합니다.

예를 들어 아래와 같은 흐름이 가능해야 합니다.

- 아기사자 객체 생성
- 이름과 역할 출력
- `doTask()` 호출
- `submitAssignment()` 호출
- 고유 기능 호출

## 예시 실행 결과
아래는 한 가지 예시입니다.

```text
철수 / 역할: 아기사자
철수님은 세션을 듣고 학습합니다.
철수님이 과제를 제출했습니다.
철수님이 질문을 합니다.

영희 / 역할: 운영진
영희님은 세션 자료를 준비하고 발표를 진행합니다.
영희님이 과제 피드백을 제공합니다.

민석 / 역할: 대표
민석님은 운영 현황을 파악하고 방향을 조율합니다.
민석님이 외부 대학과 소통합니다.
```

## 생각해볼 질문
- `BabyLion`, `Staff`, `Leader`가 `Member`를 상속받는 이유는 무엇인가?
- `name`과 `assignmentSubmitted`를 자식 클래스에 다시 만들지 않아도 되는 이유는 무엇인가?
- `getRole()`과 `doTask()`를 각 클래스에서 따로 구현하는 이유는 무엇인가?
- 상속을 사용하지 않았다면 어떤 코드 중복이 생겼을까?

## 힌트
- 자식 클래스에서 부모의 `public` 메서드는 그대로 사용할 수 있습니다.
- 예를 들어 `getName()`이나 `submitAssignment()`는 자식 클래스에서 다시 만들지 않아도 됩니다.
- `getRole()`과 `doTask()`는 반드시 오버라이딩해야 합니다.
- `@Override`를 붙이면 오버라이딩 여부를 더 안전하게 확인할 수 있습니다.
