# 🦁 멋사 공주대 백엔드 객체지향 2주차 과제 Level 1

## 과제 주제
**"멋사 구성원을 객체로 표현하며 추상화와 캡슐화의 필요성 체감하기"**

---

## 과제 목표
1. **추상화(Abstraction):** 아기사자, 운영진, 대표를 `Member`라는 공통 개념으로 묶기
2. **캡슐화(Encapsulation):** 필드(상태)를 외부의 무분별한 수정으로부터 보호하기

---

## 상황 설명
멋사에는 다양한 역할을 가진 구성원들이 존재합니다. 이들은 각자 하는 일은 다르지만, 모두 "멋사의 구성원"이라는 공통점을 가집니다.

| 역할 | 설명 |
| :--- | :--- |
| **아기사자** | 세션을 듣고 학습하며 과제를 제출하는 사람 |
| **운영진** | 세션 자료를 준비하고 발표 및 피드백을 하는 사람 |
| **대표** | 동아리 운영을 조율하고 외부와 소통하는 사람 |

---

## 구현 요구사항


### 1. 추상 클래스 사용
`Member`는 객체로 바로 생성할 수 없는 **추상 클래스**로 구현하세요.
> **이유:** “구성원”은 공통 개념이지, 그 자체로 구체적인 객체는 아니기 때문입니다.

### 2. 캡슐화 적용
아래 두 필드는 캡슐화를 적용하기 위한 접근제어자를 선택해서 선언하세요.
* `name`
* `assignmentSubmitted`

**직접 접근 제한 예시:**
```java
// 아래와 같은 직접 접근은 불가능해야 합니다.
member.name = "김기현";
member.assignmentSubmitted = true;
```
대신 필요한 경우에는 메서드를 통해서만 상태를 다룰 수 있어야 합니다.

### 3. 이름 검증
생성자에서 이름이 아래 조건 중 하나이면 예외를 발생시키세요.
* `null`
* 빈 문자열 `""`
* 공백만 있는 문자열 `"   "`

**예외 처리:**
* `IllegalArgumentException`을 사용하세요.
* **예시:** `throw new IllegalArgumentException("이름은 비어 있을 수 없습니다.");`

### 4. 과제 제출 로직
`submitAssignment()` 메서드는 다음과 같이 동작해야 합니다.

* **아직 제출하지 않았다면**
    * `assignmentSubmitted`를 `true`로 변경
    * `"<이름>님이 과제를 제출했습니다."` 출력
* **이미 제출한 상태라면**
    * 상태를 바꾸지 않고
    * `"<이름>님은 이미 과제를 제출했습니다."` 출력

### 5. 역할과 책임은 강제
아래 두 메서드는 `Member`에서 직접 구현하지 않고, 나중에 자식 클래스가 반드시 구현하도록 **추상 메서드**로 선언하세요.
* `getRole()`
* `doTask()`

## 생각해볼 질문
1. **왜 `Member`는 일반 클래스가 아니라 추상 클래스인가?**
   1. 아래에 추상 메서드를 선언했으니, 추상 클래스여야함.
   2. 추상 메서드가 선언되었는데 일반 클래스로 사용 시, 오류가 발생하기 때문에 추상 메서드를 포함하고 있는 추상 클래스라고 표시하는 것
2. **왜 `name`과 `assignmentSubmitted`를 `private`으로 두어야 하는가?**
    1. `public` 으로 두면  외부에서 직접 접근해 수정이 가능함. 이는 캡슐화가 깨짐.
3. **`submitAssignment()` 없이 필드를 직접 수정하게 하면 어떤 문제가 생기는가?**
    1. 중복의 제출이 발생할 수도 있고, 데이터 무결성이 훼손됨.
4. **지금 단계에서 아직 자식 클래스를 만들지 않았는데도 `getRole()`과 `doTask()`를 미리 선언한 이유는 무엇인가?**
    1. 자식 클래스에서 꼭 구현되어야하는 메서드이기 때문에 미리 선언해두어서 자식 클래스에서 강제로 생성하게 함.
